# Feature Specification: 比特币交易平台 MVP

**Feature Branch**: `001-mvp`  
**Created**: 2025-10-14  
**Status**: Draft  
**Input**: 用户描述: "开发一个可部署的、功能完整的 比特币交易平台 MVP 版本，核心功能包括：用户注册与登录、实时行情、基础下单与挂单功能（模拟撮合）、用户资产管理（虚拟钱包）、交易记录展示、响应式网页界面（PC + 移动端）"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 账户创建与首次登录 (Priority: P1)

新用户访问平台，通过邮箱完成注册，获得初始虚拟资产（1.0 BTC 和 20,000 USDT），并成功登录查看账户信息。

**Why this priority**: 这是所有功能的基础。没有用户账户，用户无法使用任何交易功能。这是最小可用产品的核心前提。

**Independent Test**: 用户可以独立完成从注册到登录的完整流程，无需其他功能支持。测试完成后用户可以看到个人资产余额。

**Acceptance Scenarios**:

1. **Given** 用户未注册, **When** 用户输入有效邮箱和密码并提交注册表单, **Then** 系统创建账户并自动分配初始资产（1.0 BTC + 20,000 USDT）
2. **Given** 用户已注册, **When** 用户输入正确的邮箱和密码登录, **Then** 系统显示用户个人中心，包含当前资产余额
3. **Given** 用户已登录, **When** 用户查看个人资料页面, **Then** 显示用户邮箱和完整的资产列表（BTC 和 USDT 余额）

---

### User Story 2 - 实时行情查看 (Priority: P1)

用户登录后，可以在主界面看到 BTC/USDT 交易对的实时价格、24小时涨跌幅和成交量数据。

**Why this priority**: 实时行情是交易决策的基础，用户需要看到市场数据才能做出交易决定。这是核心交易体验的第一步。

**Independent Test**: 无需进行任何交易操作，用户即可查看和验证行情数据的实时更新。可以通过对比第三方数据源验证准确性。

**Acceptance Scenarios**:

1. **Given** 用户已登录, **When** 用户进入交易主页, **Then** 页面顶部显示当前 BTC/USDT 价格
2. **Given** 行情数据正在展示, **When** 市场价格发生变化, **Then** 页面上的价格、涨跌幅和成交量数据自动更新（无需刷新页面）
3. **Given** 用户查看行情, **When** 价格上涨, **Then** 涨跌幅以绿色显示；**When** 价格下跌, **Then** 涨跌幅以红色显示
4. **Given** 用户在移动设备上访问, **When** 用户查看行情, **Then** 行情数据以响应式布局正确显示，适配小屏幕

---

### User Story 3 - 提交限价单交易 (Priority: P2)

用户可以设置目标价格和数量，提交买入或卖出的限价单，系统记录该订单并尝试与市场上的反向订单进行撮合。

**Why this priority**: 限价单是最基础的交易类型，允许用户以特定价格下单。这是实现交易功能的核心能力。

**Independent Test**: 用户可以提交限价单并查看订单列表，验证订单是否正确记录。可独立于其他用户的订单测试。

**Acceptance Scenarios**:

1. **Given** 用户持有足够的 USDT, **When** 用户输入买入价格、数量并提交限价买单, **Then** 系统扣除相应的 USDT（价格 × 数量），订单出现在"我的订单"列表
2. **Given** 用户持有足够的 BTC, **When** 用户输入卖出价格、数量并提交限价卖单, **Then** 系统锁定相应的 BTC，订单出现在"我的订单"列表
3. **Given** 限价单已提交, **When** 市场上存在满足价格条件的反向订单, **Then** 系统自动撮合成交，更新双方资产余额，订单状态变为"已成交"
4. **Given** 限价单部分成交, **When** 剩余数量继续等待, **Then** 订单显示为"部分成交"状态，未成交部分继续挂单

---

### User Story 4 - 提交市价单交易 (Priority: P2)

用户希望立即以当前市场价格买入或卖出，系统按最优可用价格立即执行交易。

**Why this priority**: 市价单提供快速成交能力，适合希望立即完成交易的用户。这是交易灵活性的重要补充。

**Independent Test**: 用户提交市价单后，可立即看到成交结果和资产变化，无需等待撮合。

**Acceptance Scenarios**:

1. **Given** 用户持有足够的 USDT, **When** 用户输入买入数量并提交市价买单, **Then** 系统以当前最优卖价立即成交，扣除相应 USDT 并增加 BTC
2. **Given** 用户持有足够的 BTC, **When** 用户输入卖出数量并提交市价卖单, **Then** 系统以当前最优买价立即成交，扣除 BTC 并增加 USDT
3. **Given** 市价单成交, **When** 交易完成, **Then** 交易记录立即出现在"我的成交"列表中

---

### User Story 5 - 查看订单簿和成交历史 (Priority: P3)

用户可以查看当前市场上所有挂单的价格和数量（买盘和卖盘），以及最近的成交记录，帮助理解市场深度和趋势。

**Why this priority**: 订单簿和成交历史提供市场透明度，帮助用户做出更明智的交易决策，但不是交易的必要前提。

**Independent Test**: 用户可以独立浏览订单簿和成交历史，无需进行任何交易操作。

**Acceptance Scenarios**:

1. **Given** 市场存在挂单, **When** 用户查看订单簿, **Then** 显示所有买单（从高到低排序）和卖单（从低到高排序），包含价格和数量
2. **Given** 有新的交易成交, **When** 用户查看成交历史, **Then** 最新的成交记录出现在列表顶部，显示成交价格、数量和时间
3. **Given** 用户提交了挂单, **When** 用户查看"我的订单", **Then** 显示所有未成交和部分成交的订单，包含订单类型、价格、数量和状态

---

### User Story 6 - 资产余额管理 (Priority: P3)

用户可以随时查看当前的 BTC 和 USDT 余额，以及资产的实时估值（以 USDT 计价）。

**Why this priority**: 资产管理是用户了解自身财务状况的窗口，虽然重要但可以在基础交易功能之后实现。

**Independent Test**: 用户可以查看资产页面，验证余额准确性，无需进行交易操作。

**Acceptance Scenarios**:

1. **Given** 用户已登录, **When** 用户访问资产页面, **Then** 显示 BTC 和 USDT 的可用余额和冻结余额
2. **Given** 用户有挂单, **When** 订单等待成交, **Then** 相应的资产显示为"冻结"状态
3. **Given** 用户查看资产, **When** BTC 价格变化, **Then** 总资产估值（以 USDT 计价）自动更新

---

### Edge Cases

- **网络连接中断**: 用户在提交订单过程中网络断开，系统如何处理？订单是否会重复提交？
- **余额不足**: 用户试图提交超出可用余额的订单时，系统如何提示并阻止？
- **价格剧烈波动**: 市价单提交时，如果价格瞬间大幅跳动，用户是否会以不合理的价格成交？
- **并发订单**: 多个用户同时提交订单试图与同一挂单成交，系统如何保证公平性和数据一致性？
- **订单取消**: 用户提交挂单后，在成交前能否取消？取消后资产如何解冻？
- **异常数据源**: 如果行情数据源（如 Binance WebSocket）断开或延迟，系统如何处理？是否显示过时数据或错误提示？
- **移动端操作**: 在小屏幕设备上，复杂的订单簿和交易表单如何保持可用性？
- **会话过期**: 用户登录后长时间未操作，会话过期时是否会丢失未提交的订单草稿？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须支持用户通过邮箱和密码进行注册
- **FR-002**: 系统必须验证邮箱格式的有效性（符合标准邮箱格式规范）
- **FR-003**: 系统必须在用户注册成功后自动分配初始虚拟资产：1.0 BTC 和 20,000 USDT
- **FR-004**: 系统必须支持已注册用户通过邮箱和密码登录
- **FR-005**: 系统必须在用户登录后展示用户的个人信息（邮箱）和资产余额
- **FR-006**: 系统必须实时获取并展示 BTC/USDT 交易对的市场数据，包括：当前价格、24小时涨跌幅、24小时成交量
- **FR-007**: 系统必须在价格变化时自动更新行情显示，无需用户手动刷新页面
- **FR-008**: 系统必须支持用户提交限价买单和限价卖单，用户需指定价格和数量
- **FR-009**: 系统必须支持用户提交市价买单和市价卖单，用户仅需指定数量
- **FR-010**: 系统必须在用户提交订单前验证用户是否有足够的资产余额（买单需要 USDT，卖单需要 BTC）
- **FR-011**: 系统必须在订单提交后立即扣除或冻结相应的资产
- **FR-012**: 系统必须实现订单撮合逻辑，自动匹配价格相符的买卖订单
- **FR-013**: 系统必须在订单成交后更新双方用户的资产余额
- **FR-014**: 系统必须记录所有订单的详细信息，包括：用户ID、订单类型（限价/市价）、方向（买/卖）、价格、数量、状态（挂单/部分成交/已成交/已取消）、创建时间
- **FR-015**: 系统必须记录所有成交记录，包括：买方用户ID、卖方用户ID、成交价格、成交数量、成交时间
- **FR-016**: 用户必须能够查看自己的所有订单历史（包括挂单、已成交、已取消）
- **FR-017**: 用户必须能够查看自己的所有成交历史
- **FR-018**: 用户必须能够查看市场订单簿，显示当前所有挂单（买盘和卖盘）
- **FR-019**: 用户必须能够查看市场最近的成交记录（公共成交历史）
- **FR-020**: 用户必须能够查看自己的资产余额，包括：BTC 可用余额、BTC 冻结余额、USDT 可用余额、USDT 冻结余额
- **FR-021**: 用户必须能够查看总资产估值（以 USDT 计价，根据当前 BTC 价格计算）
- **FR-022**: 系统必须支持响应式布局，在桌面和移动设备上均可正常使用
- **FR-023**: 系统必须在用户余额不足时阻止订单提交，并显示明确的错误提示
- **FR-024**: 系统必须在订单价格或数量为非法值（如负数、零、非数字）时阻止提交，并显示错误提示
- **FR-025**: 用户必须能够取消自己尚未成交或部分成交的挂单
- **FR-026**: 系统必须在挂单取消后解冻相应的资产
- **FR-027**: 系统必须区分不同状态的订单：挂单中、部分成交、完全成交、已取消

### Key Entities

- **User（用户）**: 代表平台的注册用户，包含邮箱、密码（加密存储）、注册时间、最后登录时间
- **Asset（资产）**: 代表用户持有的虚拟资产，包含用户ID、资产类型（BTC/USDT）、可用余额、冻结余额。每个用户对每种资产类型有一条记录
- **Order（订单）**: 代表用户提交的交易订单，包含用户ID、交易对（BTC/USDT）、订单类型（限价/市价）、方向（买/卖）、价格（市价单为空）、数量、已成交数量、订单状态、创建时间、更新时间
- **Trade（成交记录）**: 代表一次成交事件，包含买方订单ID、卖方订单ID、成交价格、成交数量、成交时间。一个订单可能对应多条成交记录（部分成交场景）
- **Market Ticker（市场行情）**: 代表 BTC/USDT 交易对的实时行情数据，包含当前价格、24小时涨跌幅、24小时成交量、最高价、最低价、更新时间（数据源自外部交易所）

### Assumptions

1. **初始资产分配**: 所有新注册用户统一获得 1.0 BTC 和 20,000 USDT 作为初始虚拟资产，这是 MVP 的简化设计，无需充值功能
2. **交易对限制**: MVP 仅支持 BTC/USDT 单一交易对，不支持其他加密货币
3. **撮合逻辑简化**: 撮合引擎按价格优先、时间优先原则进行简单撮合，不考虑复杂的市场深度算法或高频交易场景
4. **市价单执行**: 市价单以当前最优价格立即成交，如果市场深度不足，订单将部分成交或拒绝（取决于实现细节）
5. **行情数据源**: 实时行情数据来源于 Binance 公开 WebSocket API，不需要 API 密钥认证
6. **模拟交易**: 所有交易均为虚拟模拟交易，不涉及真实资金或区块链交易
7. **用户隔离**: 用户只能查看和管理自己的订单和资产，不能访问其他用户的私有信息
8. **会话管理**: 用户登录后会话有效期为标准 Web 应用时长（如 7 天），超时后需要重新登录
9. **数据持久化**: 所有用户数据、订单和成交记录必须持久化存储，系统重启后数据不丢失
10. **错误处理**: 对于网络错误、数据源异常等情况，系统应显示友好的错误提示，而不是崩溃或显示技术性错误信息

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 新用户可以在 2 分钟内完成从注册到首次登录的全流程，并看到初始资产余额
- **SC-002**: 行情数据更新延迟不超过 2 秒（从数据源更新到页面显示）
- **SC-003**: 用户提交订单后，系统在 1 秒内完成验证和记录，并显示确认或错误提示
- **SC-004**: 当存在匹配的反向订单时，订单撮合在 3 秒内完成，资产余额正确更新
- **SC-005**: 用户可以在移动设备（屏幕宽度 < 768px）上正常查看行情、提交订单和查看资产，界面无布局错乱
- **SC-006**: 90% 的用户能够在首次使用时成功提交一笔订单，无需查看帮助文档
- **SC-007**: 系统在 100 个并发用户同时访问时，页面加载时间不超过 3 秒
- **SC-008**: 订单簿和成交历史数据显示准确率 100%（与数据库记录一致）
- **SC-009**: 用户资产余额在所有交易操作前后保持一致性（总资产 = 可用 + 冻结，交易前后总资产守恒）
- **SC-010**: 当用户余额不足时，100% 的订单提交尝试被系统阻止，并显示明确的错误提示
